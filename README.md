# Python24_Exam
Код реалізує систему управління даними про мешканців та квартири, яка дозволяє додавати, видаляти мешканців і квартири, а також зв'язувати мешканців із квартирами. 
Ось опис основних класів і функцій:
1. FileManager:Відповідає за завантаження та збереження даних у файл у форматі JSON.

Методи:

	load(): Завантажує дані з файлу, обробляючи помилки (наприклад, якщо файл не знайдено чи JSON некоректний).

	save(data): Зберігає дані у файл.
2. Resident:
Моделює мешканця з такими атрибутами, як ім'я, ІПН, дата народження, телефон, email, додаткова інформація та номер квартири.

Методи:

	to_dict(): Перетворює об'єкт мешканця в словник для серіалізації.

	__eq__(other): Порівнює мешканців за ІПН.
3. Apartment:
Моделює квартиру з атрибутами, такими як номер, під'їзд, кількість поверхів, номер поверху, кількість кімнат і список мешканців.

Методи:
	
add_resident(resident): Додає мешканця до квартири.

	remove_resident(tax_id): Видаляє мешканця з квартири за ІПН.
	
to_dict(): Перетворює квартиру в словник для серіалізації.
	
__eq__(other): Порівнює квартири за номером.

4. HouseRepository:
Репозиторій для зберігання даних про мешканців і квартири.

Методи:

	find_resident_by_tax_id(tax_id): Пошук мешканця за ІПН.

	find_apartment_by_number(number): Пошук квартири за номером.
	
add_resident(resident): Додає мешканця.

	remove_resident(tax_id): Видаляє мешканця.
	
add_apartment(apartment): Додає квартиру.
	
remove_apartment(number): Видаляє квартиру.

	assign_resident_to_apartment(tax_id, apartment_number): Закріплює мешканця за квартирою.

	unassign_resident_from_apartment(tax_id): Відкріплює мешканця від квартири.
5. Validator:
Клас для перевірки даних на коректність.

Методи:

	validate_phone(phone): Перевіряє правильність формату телефону.
	
validate_email(email): Перевіряє правильність формату email.
	validate_tax_id(tax_id): Перевіряє формат ІПН.
	
validate_date(date): Перевіряє формат дати.
	
validate_apartment(apartment): Перевіряє правильність формату номера квартири.

6. HouseManagementService:
Сервіс для управління мешканцями та квартирами.

Методи:
	
add_resident(): Додає нового мешканця, перевіряючи коректність введених даних.

	remove_resident(): Видаляє мешканця за ІПН.
	
add_apartment(): Додає нову квартиру.
	
remove_apartment(): Видаляє квартиру.

	assign_resident_to_apartment(): Закріплює мешканця за квартирою.
	unassign_resident_from_apartment(): Відкріплює мешканця від квартири.
	
generate_report_residents(): Виводить список усіх мешканців.

	generate_report_apartments(): Виводить список усіх квартир.
	
report_residents_by_apartment(): Виводить список мешканців по квартирах.
	
report_unassigned_residents(): Виводить список мешканців без квартир.

Код забезпечує зручне управління даними про мешканців та квартири, а також перевірку даних на коректність перед додаванням або оновленням.
Він реалізує управління даними про мешканців і квартири за допомогою об'єктно-орієнтованого програмування (ОПП) та принципів SOLID. 


1. Принцип єдиного обов'язку (Single Responsibility Principle, SRP)

Кожен клас виконує тільки одну функцію:


FileManager: відповідає за завантаження та збереження даних у файл.

Resident: представляє мешканця та має атрибути, пов'язані з ним.

Apartment: відповідає за інформацію про квартиру та її мешканців.

HouseRepository: містить логіку збереження, пошуку та оновлення даних про мешканців і квартири.

HouseManagementService: містить бізнес-логіку для управління мешканцями та квартирами (додавання, видалення, закріплення/відкріплення мешканців від квартир).

Validator: здійснює перевірку правильності введених даних (телефон, email, ІПН тощо).
Кожен клас виконує одну чітко визначену задачу, що робить систему більш зрозумілою, гнучкою та зручною для підтримки.


2. Принцип відкритості/закритості (Open/Closed Principle, OCP)

Класи закриті для змін, але відкриті для розширення. 
Це означає, що ви можете додавати нові функціональності без зміни існуючого коду:


Для того, щоб додати нові перевірки, можна створити нові методи в класі Validator, не змінюючи решту програми.

Додавати нові методи для роботи з мешканцями чи квартирами в класі HouseManagementService або HouseRepository можна без змін у вже реалізованих методах.

3. Принцип підстановки Лісков (Liskov Substitution Principle, LSP)

Цей принцип полягає в тому, що об'єкти підкласів повинні бути заміними об'єктами своїх батьківських класів.
У коді цей принцип не настільки явний, оскільки не використовуються явні підкласи. 
Проте, при необхідності можна легко додати класи-нащадки для Resident або Apartment, наприклад, для специфічних типів мешканців або квартир, не змінюючи існуючий код.


4. Принцип розділення інтерфейсу (Interface Segregation Principle, ISP)

Принцип ISP говорить про те, що не варто змушувати класи реалізовувати інтерфейси, які вони не використовують. 
У коді інтерфейсів явно не використовуються, але принцип можна застосувати через створення окремих методів для кожної конкретної функції:


HouseRepository має чітко розділені методи для роботи з мешканцями та квартирами.

HouseManagementService не перевантажений зайвими методами і має відповідні дії, без зайвих залежностей.

5. Принцип інверсії залежностей (Dependency Inversion Principle, DIP)

Цей принцип вимагає, щоб класи залежали від абстракцій, а не від конкретних реалізацій:


HouseManagementService залежить від HouseRepository, а не від конкретних реалізацій збереження даних чи валідації, 
що дозволяє легко змінювати реалізацію репозиторію (наприклад, замінити збереження в файл на роботу з базою даних).

HouseRepository залежить від FileManager, що дозволяє змінювати методи збереження та завантаження даних без зміни логіки репозиторію.
Узагальнення:

ОПП: Реалізоване чітке використання об'єктів з інкапсуляцією та методами, які відповідають за обробку та зберігання даних.

SOLID: Принципи SOLID застосовані через чітке розділення відповідальностей класів, можливість їх розширення, 
забезпечення абстракцій для зручності заміни частин системи та підтримку гнучкості для майбутніх змін.

Цей підхід робить програму масштабованою та легшою для розширення і модифікацій у майбутньому.
